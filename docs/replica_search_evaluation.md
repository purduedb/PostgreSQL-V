# Replica Search Performance Evaluation

Operator guide for running Experiment A (recall vs. step) and Experiment B
(throughput vs. time) on the physical-replication standby using the
msturing-10M sliding-window runbook.

---

## 1. Overview

Two C++ drivers measure how the read-only standby performs against the primary
during a realistic update workload on the
`msturing-10M_slidingwindow_runbook.yaml` runbook (96-dim float32, 400 ops:
steps 1-100 insert 5M vectors; steps 101-400 cycle `[search, delete, insert]`
in 100 logical rounds).

**Experiment A — Recall vs. step** (`replica_recall_eval`)

Walks the runbook sequentially on the primary. Before each search step, waits
for the standby's WAL replay to catch up, then computes average recall@100 on
both primary and standby. Output: `experiment_a.csv` with columns
`step,primary_recall,standby_recall`.

**Experiment B — Throughput vs. time** (`replica_throughput_eval`)

Two concurrent processes (one per host):

- `--role primary` runs the mixed-mode runbook and records a wall-clock
timestamp at each group boundary. Output: `step_boundaries.csv`
(`step,timestamp_ms`).
- `--role standby` hammers random queries continuously until SIGINT and records
per-second QPS. Output: `standby_throughput.csv` (`timestamp_ms,qps`).

Both CSVs express timestamps in primary-DB-clock milliseconds so they overlay
directly despite running on different hosts.

## 2. Prerequisites

### Software

- PostgreSQL 15+ built with this extension (`shared_preload_libraries='vector'`
in both primary and standby configs).
- The extension must be installed: `CREATE EXTENSION vector;` in the target
database.
- g++, libpq-dev, libyaml-cpp-dev, libssl-dev, libgomp (OpenMP) on the build
host.

## 3. Replication setup

The drivers connect to an externally-managed primary and standby via `host:port`
flags. They do NOT run `pg_ctl`, `pg_basebackup`, or edit any config files. The
operator is responsible for all PostgreSQL lifecycle and replication setup.

### Primary `postgresql.conf`

```
wal_level = replica
max_wal_senders = 8
max_worker_processes = 32
wal_keep_size = 1GB
shared_preload_libraries = 'vector'
pgvector.replication_role = 'primary'
pgvector.replication_primary_port = <P>
pgvector.replication_shared_secret = '<secret>'
pgvector.storage_base_dir = '<abs path to segment storage>'
```

### Standby `postgresql.conf`

```
hot_standby = on
max_worker_processes = 32
shared_preload_libraries = 'vector'
pgvector.replication_role = 'standby'
pgvector.replication_primary_host = '<primary ip>'
pgvector.replication_primary_port = <P>
pgvector.replication_shared_secret = '<secret>'
pgvector.replication_fetch_parallelism = 2
pgvector.storage_base_dir = '<abs path to segment storage>'
```

The `pgvector.replication_shared_secret` must match on both sides. The standby
storage dir is typically the same path value as the primary, but on the standby
machine's filesystem.

### Initial sync

1. Run `pg_basebackup` from the standby host to copy the primary's data
  directory:
2. Rsync the segment storage so the standby starts with an already-populated
  index store (optional but recommended for a cold-start-free baseline):
3. Start the standby. Confirm streaming replication is active on the primary:
  ```sql
   SELECT * FROM pg_stat_replication;
  ```

## 4. Seeding for Experiment A

Before running `replica_recall_eval`, the primary must have 5M rows and a
vector index. Choose one of the two startup modes:

### Mode A: Seed-at-101 (recommended for iteration)

Insert steps 1-100 manually (or by a previous run) and build the index once on
the primary, then let it replicate to the standby before running the eval driver.
When the standby is caught up, run:

```bash
./replica_recall_eval \
  --dataset /ssd_root/dataset/turing10m/msturing-10M.fvecs \
  --queries /ssd_root/dataset/turing10m/msturing-query.fvecs \
  --runbook ../msturing-10M_slidingwindow_runbook.yaml \
  --dataset-name msturing-10M \
  --gt-dir /ssd_root/liu4127/msturing_runbook_gt \
  --primary-host <P_IP> --primary-port <P_PORT> --primary-db postgres --primary-user <U> \
  --standby-host <S_IP> --standby-port <S_PORT> --standby-db postgres --standby-user <U> \
  --start-step 101 \
  --out experiment_a.csv
```

The driver skips steps 1-100 and starts at step 101 with the assumption that
data and index are already in place.

### Mode B: Full-from-1 (self-contained)

Create the table and extension on the primary before the run. The driver will
insert steps 1-100 itself and then build the index at the specified step:

```sql
-- Run once on the primary before starting the driver:
CREATE EXTENSION vector;
CREATE TABLE vectors (id bigint PRIMARY KEY, vec vector(96));
```

Then run:

```bash
./replica_recall_eval \
  --dataset /ssd_root/dataset/turing10m/msturing-10M.fvecs \
  --queries /ssd_root/dataset/turing10m/msturing-query.fvecs \
  --runbook ../msturing-10M_slidingwindow_runbook.yaml \
  --dataset-name msturing-10M \
  --gt-dir /ssd_root/liu4127/msturing_runbook_gt \
  --primary-host <P_IP> --primary-port <P_PORT> --primary-db postgres --primary-user <U> \
  --standby-host <S_IP> --standby-port <S_PORT> --standby-db postgres --standby-user <U> \
  --start-step 1 --build-index-before 101 \
  --out experiment_a.csv
```

The driver inserts 5M vectors (steps 1-100), builds the HNSW index at step 101,
waits for the standby to catch up, then begins the recall evaluation loop.

```

  --hnsw-m N --hnsw-ef-construction N --ivfflat-lists N  --table-name NAME  --out FILE-
```

### Catchup semantics

"Standby caught up" means WAL replay only: the driver polls the standby until
`pg_last_wal_replay_lsn() >= pg_current_wal_lsn()` (the LSN captured on the
primary just before the search step). This ensures the standby has received
all mutations; it does not wait for the SegmentFetcher to transfer flushed
segments from the primary. Correctness is preserved by redo; the fetcher is
efficiency-only.

If you want a fairer recall comparison (avoid transient recall dips caused by
segments that exist on the primary but have not yet been fetched by the
standby), add `--standby-settle-ms 2000` (or another value) to sleep briefly
after WAL catchup.

### Reading `experiment_a.csv`

Columns: `step`, `primary_recall`, `standby_recall`. One row per search step
(steps 101, 104, 107, ... 400 in this runbook; each search step corresponds to
a `[search, delete, insert]` cycle). Recall values are 6 decimal places.
Rows with a missing GT file emit `NaN` for both recalls.

- A persistent `standby_recall < primary_recall` across many steps indicates
replication divergence or a missed segment transfer.
- A transient dip near flush boundaries is expected fetch lag; raise
`--standby-settle-ms` to quantify.

## 6. Run Experiment B

### Launch order is critical: start the standby driver FIRST

The two drivers run on different hosts. Because the standby driver records
continuous QPS from when it starts, it must be running before the primary
workload begins. Follow this sequence:

**Step 1 — On the STANDBY host, start the query load** (runs until you send SIGINT):

```bash
./replica_throughput_eval --role standby \
  --queries /ssd_root/dataset/turing10m/msturing-query.fvecs \
  --standby-host 127.0.0.1 --standby-port <S_PORT> \
  --standby-db postgres --standby-user <U> \
  --primary-host <P_IP> --primary-port <P_PORT> \
  --primary-db postgres --primary-user <U> \
  --hnsw-ef-search 200 \
  --threads 16 \
  --out standby_throughput.csv
```

This opens one short-lived connection to the primary to prime the clock offset
(RTT/2-corrected; see Clock sync below), then queries only the standby for the
duration of the run.

**Step 2 — On the PRIMARY host, run the mixed-mode workload:**

```bash
./replica_throughput_eval --role primary \
  --dataset /ssd_root/dataset/turing10m/msturing-10M.fvecs \
  --queries /ssd_root/dataset/turing10m/msturing-query.fvecs \
  --runbook ../msturing-10M_slidingwindow_runbook.yaml \
  --dataset-name msturing-10M \
  --primary-host 127.0.0.1 --primary-port <P_PORT> \
  --primary-db postgres --primary-user <U> \
  --mixed-mode-start 101 --mixed-size 3 \
  --threads 16 \
  --out step_boundaries.csv
```

Note: `--queries` is required for the primary role too because each mixed group
in this runbook contains a search step.

**Step 3 — When the primary finishes**, send SIGINT (Ctrl-C) to the standby
driver. It will merge completion timestamps, bucket them by second, and write
`standby_throughput.csv`.

### Full flag reference (from `./replica_throughput_eval` with no args)

```
Usage: ./replica_throughput_eval --role primary|standby [options]
  --dataset PATH --queries PATH --runbook PATH --dataset-name NAME (primary needs all; standby needs --queries)
  --primary-host/-port/-db/-user/-pass   (both roles: clock priming; primary: workload)
  --standby-host/-port/-db/-user/-pass   (standby role: query target)
  --index-type hnsw|ivfflat --hnsw-ef-search N --ivfflat-probes N
  --mixed-mode-start N (101) --mixed-size N (3) --threads N (16)
  --checkpoint-size N (1000) --dataset-offset N (0) --duration-sec N (0=until SIGINT)
  --assume-synced-clocks  --table-name NAME --out FILE
```

### Clock sync across different hosts

Both CSVs timestamp events in primary-DB-clock milliseconds so they share one
timeline. The standby driver primes a clock offset by sampling
`clock_timestamp()` on the primary K=5 times and picking the sample with the
smallest round-trip time; the offset is `primary_ms - (t0 + rtt/2)`. The
timing error is bounded to a few milliseconds over a LAN, well under the
1-second QPS bucket width.

If the hosts are NTP-synchronized (clocks differ by less than ~10 ms), you may
skip priming on both sides:

```bash
./replica_throughput_eval --role standby ... --assume-synced-clocks
./replica_throughput_eval --role primary ... --assume-synced-clocks
```

With `--assume-synced-clocks`, local `CLOCK_REALTIME` is used directly; no
connection to the primary is made for clock priming.

### Output files


| File                     | Written by       | Columns             | Notes                                                                              |
| ------------------------ | ---------------- | ------------------- | ---------------------------------------------------------------------------------- |
| `standby_throughput.csv` | `--role standby` | `timestamp_ms,qps`  | One row per second that had >= 1 completion; primary-DB-clock epoch-ms             |
| `step_boundaries.csv`    | `--role primary` | `step,timestamp_ms` | One row per mixed group (groups of 3 ops); step = lead step number (101, 104, ...) |

## 8. Troubleshooting

### Catchup timeout (`standby catchup timeout (target …)`)

- Check `pg_stat_replication` on the primary: the `sent_lsn` and `replay_lsn`
columns show replication lag.
- Check the standby PostgreSQL log for replay errors or paused recovery.
- Increase `--catchup-timeout-sec` (default 300) if the standby is legitimately
slow to replay large batches.

### GT filename mismatch ("`MISSING GT …`" lines in stderr)

The GT filename encodes a hash of the active-ranges JSON at that step. If the
hash does not match, the driver emits `NaN` for that step. Common causes:

- `--dataset-name` does not match the name used when GT was generated. The
runbook key is `msturing-10M`; the `runbook_id` used for filenames is derived
by replacing `-` with `_` (`msturing_10M`). Pass `--dataset-name msturing-10M`
and the driver handles the conversion internally.
- `--start-step` differs from the GT generator's assumption. If GT was generated
with the full insert prefix (steps 1-100), the driver must also start from
step 1 (or use seed-at-101 mode where the prefix is already materialized).
- `--dataset-offset` is nonzero but GT was generated with offset 0 (the default).
Keep offsets consistent between GT generation and this driver.
- The GT `.npy` file for that exact step is absent from the GT directory. The
missing-GT error prints the full expected path; verify the file exists there.

### Standby serving stale results (recall lower than expected, not recovering)

- Verify that `pgvector.replication_role = 'standby'` is set and the extension
background workers are running (`SELECT * FROM pg_stat_activity WHERE backend_type LIKE '%vector%'`).
- Check `pgvector.replication_fetch_parallelism` (default may be 1); increasing
to 2-4 speeds up segment transfer.
- The SegmentFetcher is efficiency-only: the standby is correct via WAL redo
even without fetched segments, but recall may be lower until flushed segments
arrive. Use `--standby-settle-ms` in Experiment A to add a post-catchup wait.

### `--checkpoint-size` in `replica_throughput_eval`

The `--checkpoint-size` flag is accepted by `replica_throughput_eval` but is
currently unused by that tool (reserved for future use). The mixed-mode primary
driver issues single-row inserts per work item. Do not rely on it for batching
in Experiment B; it has no effect.

### Standby driver exits immediately with `cannot open …`

The `--queries` file path is required for both roles. Verify that the query
file path is accessible from the host running the standby driver (not just the
primary host).
